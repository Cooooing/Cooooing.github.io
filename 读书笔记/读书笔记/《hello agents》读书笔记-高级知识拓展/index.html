第八章 - 记忆与检索

在大语言模型（LLM）的固有设计中，模型本身是无状态（Stateless）且知识静态的。这导致智能体在复杂任务中面临“多轮对话遗忘”与“领域知识盲区”的双重局限。
本章探讨如何从认知科学中汲取灵感，为智能体构建分层记忆系统（Memory System）与检索增强生成（RAG）管道，从而赋予智能体跨越时间周期的学习能力与突破参数边界的知识获取能力。

从认知科学到智能体记忆抽象

人类的记忆并非单一的存储桶，而是一个对信息流进行分级过滤、存储与重构的复杂层级网络。认知心理学的记忆模型为智能体系统架构提供了完美的蓝图。

人类记忆机制的启发映射

  • 感觉记忆（Sensory Memory）： 持续时间极短，映射到智能体中相当于对传感器输入流（如原始音视频流、原始网络请求）的瞬时缓存。
  • 工作记忆（Working Memory）： 容量有限且生命周期短，负责当前任务的实时处理。映射为智能体在单次对话或单次任务中的上下文窗口状态（Context Window）。
  • 长期记忆（Long-term Memory）：
    • 情景记忆（Episodic Memory）： 记录特定时间、地点的个人经历。映射为智能体对过去具体交互事件、历史会话日志的序列化存储。
    • 语义记忆（Semantic Memory）： 记录抽象的概念、规则与事实。映射为智能体对用户偏好、世界常识、业务规则的结构化沉淀（如知识图谱）。

LLM 原生局限性剖析

智能体引入记忆与检索模块，是为了解决大模型底层的两大根本性缺陷：

  1. 动态状态缺失： 由于上下文窗口长度的物理限制（Token Limit），长对话中的早期关键指令必然会被截断丢弃；且模型无法跨会话（Cross-session）留存对特定用户的个性化认知。
  2. 静态参数化知识局限： 大模型的知识权重在预训练完成后即被冻结（Knowledge Cut-off），无法感知实时资讯，且在垂直专业领域（如企业内部规范）缺乏知识深度，极易产生事实性幻觉（Hallucination）。

记忆与检索架构系统设计

在框架的底层架构中，记忆（Memory）与检索（RAG）不应作为特殊的特权模块，而应被降维抽象为标准的“工具（Tool）”，由核心大模型通过统一的执行接口进行调度。

记忆系统（Memory System）的多层架构

记忆系统应采用四层解耦架构，以保障数据流转的高效与扩展性：

  1. 基础设施层： 包含核心的管理器（Manager），负责记忆的编码（Encoding）、存储路由、生命周期调度与统一接口暴露。
  2. 记忆类型层： 实现认知科学映射出的四种记忆分形（工作、情景、语义、感知）。
  3. 存储后端层： 根据数据结构异构性，混合采用多种物理引擎。包括用于语义检索的向量数据库、用于结构化持久化的关系/文档型数据库，以及用于关系推演的图数据库。
  4. 嵌入表示层（Embedding Layer）： 负责将离散文本映射为连续向量的底层算子层。

RAG 系统的流转层级

RAG 系统专注于外部静态知识库的生命周期管理，其逻辑管道分为：

  1. 文档处理层： 负责多模态或异构格式文档的解析、清洗与结构化。
  2. 分块与向量化层： 按照特定策略进行文本切片，并利用嵌入模型映射为高维张量。
  3. 智能问答层： 包含高级检索策略（如查询重写、重排）与上下文拼装模块。

四维记忆类型的底层实现机制

基于认知分形，系统中的四种记忆类型在存储介质、生命周期与检索算法上存在本质差异。

工作记忆（Working Memory）

  • 定位： 短期高频访问的上下文缓存。
  • 实现策略： 采用 纯内存（In-memory） 存储，读写延迟极低。引入 TTL（Time-to-Live，生存时间） 机制与最大容量淘汰队列。当会话结束或容量触顶时，自动清理陈旧数据。
  • 检索算法： 为追求极致速度，通常采用轻量级的 TF-IDF 等稀疏向量算法 结合关键词精准匹配，计算综合相关性得分。

情景记忆（Episodic Memory）

  • 定位： 带有明确时间戳与上下文元数据的具体事件序列。
  • 实现策略： 采用关系型/文档型数据库与向量数据库的混合（Hybrid）存储。关系型数据库保存事件的时间戳、会话 ID 与强结构化属性；向量数据库保存事件内容的语义表征。
  • 检索算法： 引入时间近因性（Recency） 衰减模型。其评分公式同时衡量：向量语义相似度、时间衰减因子（距离当前时间越近分数越高）以及事件本身被标注的重要性权重（Importance）。

语义记忆（Semantic Memory）

  • 定位： 从经历中提炼出的抽象常识、实体关系与长期规则。
  • 实现策略： 核心依赖于图数据库（如 Neo4j） 与向量库的双轨存储。在记忆写入时，系统首先利用 NLP 算法（如命名实体识别 NER 与关系抽取）将文本解析为三元组（实体-关系-实体），存入知识图谱；同时将原始文本存入向量库。
  • 检索算法： 执行图检索与向量检索的联合路由。不仅计算语义相似度，还计算图谱中的拓扑连接度（Graph Similarity），使得系统具备跳跃式关系推理（多跳查询）的能力。

感知记忆（Perceptual Memory）

  • 定位： 用于容纳图像、音频流等多模态（Multi-modal）表征。
  • 实现策略： 采用模态分离的向量空间存储。文本、图像（利用 CLIP 等模型）、音频分列在不同的向量集合中，防止高维空间的语义干扰。
  • 检索算法： 支持同模态精准检索与跨模态（Cross-modal）语义对齐检索，同样融合时间衰减与重要性权重进行联合排序。

记忆引擎的高阶管理操作

记忆系统的闭环不仅在于“存”与“查”，更在于对信息生命周期的主动干预。

  • 选择性遗忘（Forgetting）： 模拟生物大脑的代谢机制。系统支持三种维度的裁剪：
    • 阈值清理： 淘汰重要性评分（Importance）低于底线的冗余数据。
    • 时间老化： 强制清除超过特定物理时间窗口的非核心记忆。
    • 容量挤出： 触碰物理上限时，根据 LRU（最近最少使用）或重要性加权算法淘汰尾部数据。
  • 记忆整合与固化（Consolidation）：
    短期记忆向长期记忆的跃迁机制。系统可以设置后台守护进程，定期扫描工作记忆中那些“高频被访问”或“重要性极高”的临时切片，将其打包提升为情景记忆，或者通过大模型提炼其中的抽象概念，转存入知识图谱（语义记忆）中。

检索增强生成（RAG）管道深度解析

RAG 系统将通用大模型转化为特定领域的垂直专家。其核心在于解决“如何准确找出现有知识”与“如何规避检索遗漏”。

文档预处理与智能分块（Chunking）

文档解析的核心是将混沌的非结构化数据（如 PDF 扫描件、音频）降维为标准化的 Markdown 或纯文本流。

结构感知分块： 摒弃粗暴的“按字数一刀切（Fixed-size Chunking）”，这会极大地破坏语义的完整性。
现代策略优先解析 Markdown 的标题树层次（Heading Hierarchy）或 HTML 标签结构，保证每一个文本块（Chunk）在语义边界上是相对独立且完整的，同时配合滑动重叠窗口（Overlap）以保留分块间的上下文粘连。

高级检索策略（Advanced Retrieval）

当用户查询短促、模糊或术语不匹配时，基础向量检索极易发生漏召回。系统引入多路扩展策略以兜底召回率：

  • 多查询扩展（MQE, Multi-Query Expansion）：
    利用大模型的语义泛化能力，将用户单一的原始查询（Query）改写、同义替换或拆解为 $N$ 个语义等价但字面迥异的扩展查询。
    系统利用这 $N$ 个变体并行对向量库发起检索，最后对召回的候选文档集进行并集去重。这极大缓解了用户用词不专业导致的召回失败。
  • 假设文档嵌入（HyDE, Hypothetical Document Embeddings）：
    解决“疑问句与陈述句在向量空间分布不均”的根本性难题。系统首先命令大模型：“请不要去检索，仅凭你的幻觉或内置知识，直接编造一段能回答该问题的文本（假答案）”。
    随后，系统将这段“包含丰富术语的陈述句假答案”进行向量化，再去知识库中匹配真实文档。这种“用答案找答案”的逆向工程，能跨越巨大的语义鸿沟。
  • 统一合并路由：
    高级 RAG 管道通常同时执行原始查询、MQE 变体查询与 HyDE 假设查询。将所有并发召回的候选文档扔进一个巨大缓冲池，利用交叉注意力模型或倒数秩融合（RRF）算法进行重排（Rerank），截取最高分的 Top-K 输入给最终的生成模型。

课后习题解析

记忆类型与认知机制的评分推演

【原问题】

  1. 为什么在底层评分公式中，情景记忆严重依赖“时间近因性（Recency，权重 0.2）”，而语义记忆则更强调“图检索（权重 0.3）”？这种参数分配背后的认知哲学是什么？
  2. 假设设计一个“个人健康管理助手”（需要记录用户的饮食、运动、睡眠数据，并提供健康建议），你会如何将工作、情景、语义、感知这四类记忆组合应用？
  3. 在工作记忆中，哪些触发条件应当激活 consolidate（记忆固化）操作，将其转化为长期记忆？

【解析】

  1. 评分机制的认知哲学：
    • 情景记忆的本质是“时间线上的切片”。 当用户问“我昨天说过什么”时，昨天（时间点）的优先级绝对高于聊天的具体内容（语义）。因此，时间衰减（Recency）必须在评分机制中占据极高的权重，以保证最新的事件或明确指定时间戳的事件能够穿透语义相似度的压制被优先召回。
    • 语义记忆的本质是“脱水的客观真理”。 “地球是圆的”这句话无论是一年前学到的还是刚刚学到的，其价值是等同的（无时间衰减）。但语义记忆存在大量的抽象关联，比如“狗”与“柯基”，仅仅依靠向量空间可能距离较远，但通过知识图谱的本体树（Ontology Tree）可以判定为包含关系。因此，引入高权重的图检索（Graph Retrieval）是为了补足纯向量模型在严格逻辑与本体关系推理上的短板。
  2. “个人健康管理助手”的四维记忆矩阵设计：
    • 感知记忆： 接收用户每日上传的午餐照片、体检报告扫描件（PDF）、心电图波形（时序数据）。
    • 工作记忆： 缓存在当前“挂号咨询”或“症状问诊”这一特定会话周期内，用户描述的突发头痛部位、严重程度。一旦问诊结束出具建议，该部位剧痛作为临时状态即被清理出高频内存。
    • 情景记忆： 记录用户“在 2023年5月1日 进行了阑尾炎手术”、“连续三天（上周一到周三）跑步达到 5 公里”。这是严格绑定时间轴的历史事件记录。
    • 语义记忆： 从长期的历史中提炼出的绝对规则。例如：“该用户对青霉素过敏”、“用户属于易胖体质”、“用户具有家族高血压史”。这是系统在任何时刻做出医疗建议前必须被检索的最高优先级“公理”。
  3. 工作记忆整合（Consolidate）的触发阈值设计：
    整合机制不应是随机的，应当基于 “重要性-频次”双轨阈值：
    • 高重要性突发触发： 当工作记忆中出现被模型标记为极高重要性（$Importance > 0.9$）的数据（如用户突发声明：“我对花生过敏”），应立即触发硬中断，强制将其固化至语义记忆库。
    • 高频共现触发： 系统后台维护一个滑动窗口统计。若某项较小的信息（如用户总是询问“低糖食谱”）在多个独立的会话周期内被高频写入工作记忆，表明这已成为用户的长期偏好，触发批处理任务，将其抽象聚合后写入长期记忆。

RAG 系统智能分块与检索策略剖析

【原问题】

  1. Markdown 分块策略依赖标题层次。对于没有明显标题结构的长文本（如小说或法律长卷），应如何设计基于“语义边界”的智能分块算法？
  2. 试对比基础向量检索、MQE（多查询扩展）与 HyDE（假设文档嵌入）在“医疗知识检索”场景中的召回表现及适用边界。
  3. 在评估 Embedding 方案（云端 API、本地深度学习模型、TF-IDF 等）时，应从哪些架构维度进行权衡决策？

【解析】

  1. 无结构文本的语义边界分块（Semantic Chunking）：
    当无法依赖 Markdown/HTML 标签时，必须从自然语言的固有规律切入。
    • 统计学边界判定： 利用 NLTK 或 spaCy 等自然语言处理库，严格按照句子（句号/感叹号）或自然段落（双换行符）进行硬切分，严禁在句子中途斩断。
    • 动态语义凝聚（Dynamic Semantic Cohesion）： 将文本按最小单位（如单句）初始切分并向量化。依次计算相邻两个句子向量的余弦相似度。如果相似度发生断崖式下跌，说明此处发生了“话题转移”，以此下跌点作为物理切块的切分点。这种算法能够自适应地生成长短不一但内部语义绝对高内聚的文本块。
  2. 高级检索策略在垂直领域的性能博弈（以医疗为例）：
    • 基础检索： 仅适合用户的 Query 极其标准、术语精确的情况（如“阿司匹林的禁忌症有哪些”）。对普通患者模糊的口语化提问极其脆弱。
    • MQE（多查询扩展）： 极其适合症状描述发散的场景。用户输入“早上起来感觉天旋地转”，大模型会将其重写扩展为“眩晕症”、“耳石症”、“梅尼埃病表现”等多个专业学术查询。这能成倍扩大召回的命中面。
    • HyDE（假设文档嵌入）： 极其适合医理推断或机制解释的场景。用户问“为什么高血压会引起肾衰竭？”，大模型先凭已有知识生成一篇包含“肾小球动脉硬化”、“微血管病变”等密集专业词汇的假科普文。用这篇假文去匹配知识库中的权威医学指南，在向量空间中几乎是“降维打击”，其召回的精确度远超单纯用疑问句去检索。
  3. Embedding 引擎架构选型维度：
    • 业务精度（Accuracy）： 稠密模型（如 OpenAI text-embedding-3-large / text-embedding-3-small）对语义上下文的理解更强；TF-IDF 等稀疏矩阵主要做字面词频统计，在同义词泛化上能力有限。
    • 数据合规与私有化（Privacy & Compliance）： 若为军工或极度敏感数据，绝不允许将明文发往云端 API，必须强制选用本地部署的开源模型（如 BAAI/bge 系列），甚至退化至纯本地计算的 TF-IDF。
    • 延迟与基础设施成本（Latency & TCO）： 云端 API 具备极高的并发上限且无需维护本地 GPU 集群，适合轻资产团队；若在无网边缘设备（IoT）上，只能运行极度量化的微型 Transformer 嵌入模型。

多租户（Multi-Tenant）数据隔离与系统级设计

【原问题】

  1. 假设该智能体被部署为一个面向数十万注册用户的 SaaS 服务，必须保障不同用户的记忆与知识库绝对不可见。从底层数据库架构角度，应如何在向量数据库（如 Qdrant）和图数据库（如 Neo4j）中设计硬隔离方案？
  2. 在智能体的决策流中，存在同时调用 RAG（公共知识）与 Memory（个人记忆）的情况。应如何设计一种“智能路由”机制，以减少不必要的双重检索消耗？

【解析】

  1. 多租户系统下的硬性数据隔离架构（Tenant Isolation）：
    • 向量数据库（Qdrant/Pinecone）隔离： 绝对不能为每个用户创建一个独立的 Collection，这会导致底层内存耗尽。应当采用池化集合+元数据硬路由（Pooled Collections with Metadata Filtering）。将所有用户的向量塞入同一个巨大的 Collection，但在插入数据的 Payload / Metadata 中强制绑定 tenant_id（租户唯一标识）。在查询层（Query Engine）进行拦截，强制在所有的查询条件前加上强过滤约束
      Filter(must: [{key: "tenant_id", match: {value: current_user}}])，在底层物理计算距离前直接剔除无关数据。
    • 图数据库（Neo4j）隔离： 知识图谱的隔离更为复杂，因为关系可能发生交错。应采用标签与属性双轨隔离。每个实体节点必须打上对应用户的专属 Label 或持有 owner_id 属性。在执行 Cypher 查询语句时，所有的 MATCH 语法必须在应用程序的 ORM 映射层强制织入对 owner_id 的匹配子句，切断跨用户的图遍历路径。
  2. 检索意图的智能路由降维机制（Intent-based Routing）：
    并行发起对 RAG 和 Memory 的检索是极大的 I/O 浪费。应在主干链路上前置一个极速的意图分类路由（Semantic Router）或利用轻量级 LLM 节点：
    • 若用户查询包含强指向第一人称的代词或历史指代（如“我昨天说的那个项目”、“根据我以往的口味”），分类器将路由引向 Memory Retrieval（私有记忆召回）。
    • 若查询涉及客观真理、外部流程或通用的专业解释（如“解释一下量子力学”、“公司的报销流程是什么0”），分类器将路由引向 RAG Retrieval（公域知识召回）。
    • 仅在分类置信度极低（模糊地带）时，才触发并发混合检索（Hybrid Retrieval）并做倒数秩融合（RRF）重排。

第九章 - 上下文工程

提示工程关注“如何写好当前这一段 Prompt”，上下文工程关注“如何为模型构造一个足够完整、干净、可执行的任务现场”。

对智能体而言，上下文不是一段静态文本，而是由系统指令、用户目标、历史对话、工具结果、检索材料、任务状态和输出约束共同组成的运行环境。
模型的推理质量，很大程度上取决于这些信息是否被正确选择、组织和压缩。

上下文工程的核心问题

上下文工程要解决的不是“塞入更多内容”，而是把正确的信息放在正确的位置。

上下文窗口是有限资源。无关信息越多，模型越容易偏离任务；关键信息缺失，模型又会只能依赖猜测。
因此，一个可用的上下文系统必须同时处理召回率、精确率和结构稳定性。

有效上下文的组成

一个面向智能体的上下文，通常至少包含以下几类信息：

  • 任务目标： 当前要完成什么，验收条件是什么。
  • 系统约束： 哪些行为允许，哪些行为禁止，例如权限、格式、风格和安全边界。
  • 短期状态： 当前任务已经执行到哪里，最近的工具调用返回了什么。
  • 长期记忆： 用户偏好、历史结论、长期规则和反复出现的问题。
  • 外部知识： 通过 RAG、搜索、文件系统或数据库检索得到的材料。
  • 输出契约： 最终结果需要满足的结构、字段、语言和粒度。

这些信息不能简单拼接。
系统指令和安全约束应当稳定靠前，任务状态应当尽量明确，检索材料需要带来源和范围，工具返回结果要避免污染主任务。

从 Prompt 到 Context

Prompt 更像单次请求的表达方式，Context 则是智能体的运行视图。

一个长程任务中，模型每一步看到的内容都可能不同：

  • 第一轮需要理解目标和约束。
  • 中间阶段需要看到执行进度、阻塞点和已验证结论。
  • 收尾阶段需要看到验收标准、变更摘要和残留风险。

上下文工程的价值在于：每一轮都根据任务阶段重新构造输入，而不是让历史对话自然堆积。

GSSC 流水线

Hello Agents 中的 ContextBuilder 将上下文构建抽象为 GSSC 流水线。
这个模型可以理解为四个连续动作：Gather、Select、Structure、Compress。

Gather：收集候选信息

Gather 阶段负责收集所有可能有用的信息，包括对话历史、记忆系统、RAG 检索结果、工具输出和临时笔记。

这一阶段强调召回，不急于判断哪些内容一定会进入最终上下文。
设计上更像“建立候选池”，避免过早丢弃后续推理需要的信息。

Select：选择有效信息

Select 阶段负责过滤。
它需要根据当前任务目标，剔除低相关、过期、重复或可信度不足的信息。

常见选择依据包括：

  • 与当前问题的语义相关度。
  • 信息产生的时间和任务阶段。
  • 来源可信度，例如用户明确指令高于模型猜测。
  • 信息类型优先级，例如约束高于背景材料。

这一阶段的难点在于，不能只按向量相似度排序。
有些约束虽然文本相似度不高，但对任务有硬性影响。

Structure：组织上下文结构

Structure 阶段决定信息如何排列。

对模型来说，同样的信息以不同顺序出现，会产生不同的行为倾向。
通常应将稳定约束、任务目标、当前状态、参考材料和输出要求分区放置，降低模型误解的概率。

一个较稳妥的结构是：

  1. 系统角色与边界。
  2. 当前任务目标。
  3. 已知事实和任务状态。
  4. 可引用材料。
  5. 工具调用结果。
  6. 输出要求。

这种结构的核心不是形式美观，而是让模型能区分“必须遵守的约束”和“可参考的信息”。

Compress：压缩长程信息

Compress 阶段处理上下文窗口不足的问题。

压缩不等于简单摘要。
对智能体而言，压缩必须保留决策所需的信息，包括已经验证的事实、失败过的路径、用户明确要求、当前阻塞点和后续动作。

较差的压缩会把“过程”压成“结论”，导致智能体重复尝试已经失败的方案。
较好的压缩会保留关键证据和判断依据。

NoteTool 与 TerminalTool

第九章中两个工具很关键：NoteTool 负责保存结构化笔记，TerminalTool 负责即时访问文件系统。
这两个工具共同解决长程任务中的“状态保真”问题。

NoteTool：结构化任务记忆

NoteTool 的意义不是替代 Memory，而是沉淀任务过程中的显式状态。

例如，在代码库维护任务中，智能体需要记录：

  • 已经检查过哪些目录。
  • 发现了哪些阻塞点。
  • 哪些结论已经被测试验证。
  • 哪些变更需要后续处理。

这些内容不适合全部放入对话历史，也不适合只依赖模型内部记忆。
结构化笔记可以在后续轮次中重新进入 ContextBuilder，成为可检索、可压缩、可审计的上下文材料。

TerminalTool：即时探索能力

TerminalTool 的定位是让智能体按需读取真实环境，而不是预先把整个代码库、日志目录或数据文件全部索引进上下文。

它适合处理以下场景：

  • 查看项目结构。
  • 搜索函数定义或错误日志。
  • 读取配置文件。
  • 统计数据文件行数或字段。
  • 验证某个判断是否与真实文件一致。

这个工具必须配合安全机制使用。
例如限制工作目录、限制命令白名单、限制执行时间、限制输出大小，避免智能体通过工具获得过大权限或制造不可控副作用。

长程智能体的上下文维护

长程智能体最容易失败的地方，不是单步推理能力不足，而是任务状态在多轮执行中逐渐失真。

代码库维护助手就是典型场景。
它可能需要跨越数天执行探索、分析、修复、测试和总结。
在这个过程中，系统不能只依赖聊天历史，因为聊天历史会越来越长，且包含大量已经失效的中间信息。

更合理的做法是：

  • 用 TerminalTool 获取当前真实文件状态。
  • 用 NoteTool 保存阶段性发现和阻塞点。
  • 用 Memory 保存跨任务复用的长期偏好和稳定事实。
  • 用 ContextBuilder 在每一轮重新构造当前任务视图。

这样，智能体看到的是经过选择和压缩后的任务现场，而不是一整段混杂的历史记录。

课后习题解析

上下文压缩的边界

【原问题】

长程任务中，如何压缩上下文才不会丢失关键任务状态？

【解析】

上下文压缩必须围绕“后续决策是否还需要这条信息”来判断。
仅仅把历史压缩成自然语言摘要，往往会丢掉失败路径、约束来源和证据链。

较合理的压缩结果应保留：

  • 当前目标和验收标准。
  • 用户明确提出的硬性要求。
  • 已经完成并验证的动作。
  • 已经尝试但失败的方案。
  • 当前阻塞点及其证据。
  • 后续可执行动作。

压缩后的上下文应当能支持另一个智能体接手任务。
接手者必须重新阅读全部历史才能继续执行时，说明压缩质量不够。

工具结果如何进入上下文

【原问题】

工具返回结果是否应该全部放入上下文？

【解析】

不应该。
工具结果是原始材料，直接全部写入会浪费上下文窗口，还会引入大量噪声。

更稳妥的方式是分层处理：

  1. 原始结果保存在外部存储或工具日志中。
  2. 当前轮次只放入与任务有关的片段。
  3. 对长输出生成结构化摘要。
  4. 对关键结论保留来源，例如文件路径、命令和行号。

这样既能避免上下文膨胀，又能在需要验证时回到原始材料。

第十章 - 智能体通信协议

智能体系统一旦从单体演示走向真实工程，就必须处理外部工具、远程服务和其他智能体的协作问题。
通信协议的价值在于提供统一的能力描述、发现方式和调用约定，减少重复适配成本。

第十章重点介绍三类协议：MCP、A2A、ANP。
它们解决的问题不同，不能混为一谈。

协议要解决什么

没有协议时，智能体接入一个新服务，通常需要手写工具定义、参数映射、错误处理和返回值解析。
当服务数量增加后，这种方式会迅速变成高维护成本的点对点集成。

协议的目标是把集成过程标准化：

  • 服务能声明自己有哪些能力。
  • 客户端能发现这些能力。
  • 调用参数有明确结构。
  • 返回结果有统一格式。
  • 错误和权限可以被一致处理。

这相当于把“智能体会不会用工具”的问题，从模型提示词层面下沉到工程接口层面。

MCP：连接工具与资源

MCP（Model Context Protocol）关注的是模型如何连接外部工具、资源和提示模板。

它的核心价值是统一工具接入方式。
一个 MCP Server 可以向客户端暴露工具列表、参数模式、资源列表和提示模板。
智能体不需要为每个服务单独编写适配逻辑，而是通过 MCP Client 进行发现和调用。

MCP 的工程边界

MCP 适合处理“智能体调用外部能力”的场景，例如：

  • 文件系统读写。
  • 数据库查询。
  • GitHub 仓库检索。
  • 浏览器操作。
  • 企业内部系统接口。

MCP 不直接解决多智能体之间的职责分工。
它更像工具层协议，而不是协作层协议。

传输方式的差异

MCP 可以通过多种传输方式连接服务：

  • Memory Transport： 适合测试和本地演示。
  • Stdio Transport： 适合本地工具和命令行服务。
  • HTTP / SSE / Streamable HTTP： 适合远程服务和持续输出。

传输方式改变的是连接形态，不改变 MCP 的核心抽象。
真正稳定的部分是工具描述、参数结构和调用协议。

A2A：智能体之间的协作

A2A（Agent-to-Agent）关注的是智能体之间如何通信。

当一个系统中存在多个具备不同职责的智能体时，单个协调者需要能够调用研究员、代码分析员、文档生成器、客服顾问等不同角色。
A2A 的目标是让这些智能体以服务形式暴露能力，并通过统一请求进行协作。

A2A 的适用场景

A2A 适合处理以下问题：

  • 一个智能体无法独立覆盖所有专业能力。
  • 不同智能体需要独立部署、独立扩缩容。
  • 某些智能体需要持有自己的状态和工具集。
  • 协调者需要把任务拆给多个执行者。

例如，一个“接待员智能体”可以根据用户问题，把技术问题转给技术专家智能体，把销售问题转给销售顾问智能体。
这里的关键不是工具调用，而是任务委派。

ANP：网络化服务发现

ANP 更关注服务发现和智能体网络。

在一个动态网络中，协调者未必预先知道有哪些智能体可用。
ANP 的目标是让智能体能够注册能力、声明负载、按类型检索，并在运行时选择合适的服务节点。

它更接近服务治理问题：

  • 服务注册。
  • 能力发现。
  • 节点选择。
  • 负载均衡。
  • 网络关系构建。

当智能体数量较少、服务关系固定时，ANP 的收益不明显。
当智能体生态变大，或者服务需要动态加入和退出时，ANP 的价值才会体现出来。

三类协议的边界对比

协议 主要问题 核心对象 典型场景
MCP 模型如何使用外部工具和资源 工具、资源、提示模板 文件系统、数据库、第三方 API
A2A 智能体如何互相调用和协作 智能体服务 多角色协同、任务委派
ANP 智能体服务如何发现和组织 网络节点、能力注册 动态服务发现、负载选择

这三者不是互斥关系。
一个智能体可以通过 MCP 使用工具，通过 A2A 调用另一个智能体，再通过 ANP 发现当前可用的智能体服务。

自定义 MCP Server

第十章最后落到自定义 MCP Server。
这一步的意义在于：企业内部能力不可能全部由社区现成服务器提供，很多工具必须由团队自行封装。

一个自定义 MCP Server 通常包含：

  • 工具函数定义。
  • 参数 Schema。
  • 工具注册。
  • 错误处理。
  • 运行时传输方式。
  • 权限控制和部署配置。

封装内部系统时，不能只关注“能不能调通”。
还要明确哪些能力可以暴露给智能体，哪些参数需要校验，哪些操作需要人工确认，哪些返回值需要脱敏。

课后习题解析

协议选型

【原问题】

什么时候选择 MCP、A2A 或 ANP？

【解析】

先判断问题属于哪个层次。

如果问题是“智能体如何调用一个外部工具”，优先考虑 MCP。
例如读取文件、查询数据库、调用 GitHub、访问企业内部接口。

如果问题是“一个智能体如何委派任务给另一个智能体”，优先考虑 A2A。
例如协调者把调研任务分配给研究员智能体，再把结果交给写作智能体整理。

如果问题是“智能体如何发现当前网络中有哪些可用服务”，才考虑 ANP。
例如多个计算节点动态注册，调度智能体根据能力和负载选择节点。

MCP 的安全边界

【原问题】

把文件系统或数据库暴露为 MCP 工具时，需要注意什么？

【解析】

MCP 统一了工具调用方式，但不会自动消除工具风险。
安全边界必须由工具实现和运行环境共同保证。

关键控制点包括：

  • 限制可访问目录或数据表。
  • 对写操作、删除操作和外部请求设置白名单。
  • 对参数做类型校验和范围校验。
  • 对返回结果做脱敏处理。
  • 保留调用审计日志。
  • 对高风险操作引入人工确认。

模型生成的工具调用不能天然视为可信输入。
所有工具参数都应当按外部输入处理。

第十一章 - Agentic-RL

第十一章从训练角度讨论智能体能力的提升。
普通 LLM 训练更关注单轮文本生成质量，Agentic-RL 关注模型在任务环境中的行动策略。

智能体不是只生成答案，还需要选择工具、分解任务、根据反馈修正行动。
因此，训练目标不能只看最终文本是否自然，还要看行动序列是否有效。

从强化学习到 Agentic-RL

强化学习的基本结构包括状态、动作、奖励和策略。

映射到智能体场景中：

  • 状态： 当前上下文、任务进度、工具返回结果。
  • 动作： 生成回答、调用工具、请求更多信息、终止任务。
  • 奖励： 任务是否成功、答案是否正确、步骤是否高效、工具调用是否合规。
  • 策略： 模型在不同状态下选择动作的倾向。

Agentic-RL 的关键是把“会说”扩展为“会做”。
训练不只优化单次输出，而是优化多步行动后的任务结果。

LLM 训练链路

大模型能力通常来自多个阶段。

预训练让模型获得语言模式和通用知识；SFT（监督微调）让模型学习指令响应格式；偏好优化和强化学习则让模型更符合任务目标和人类偏好。

在智能体训练中，SFT 和 RL 的关系可以这样理解：

  • SFT： 教模型模仿标准答案或标准轨迹。
  • RL： 让模型根据奖励信号探索更优行动。

SFT 更稳定，但受限于标注数据。
RL 能优化最终目标，但奖励设计和训练稳定性更难。

数据集与奖励函数

第十一章使用 GSM8K 数学推理数据集作为示例。
这个选择主要是因为数学题答案可验证，适合作为奖励函数的基础。

智能体训练中，数据集通常需要区分两种格式：

  • SFT 格式： 输入问题和标准回答，用于监督学习。
  • RL 格式： 输入问题、可采样回答和奖励计算逻辑，用于强化学习。

奖励函数设计

奖励函数决定模型会朝什么方向优化。

常见奖励包括：

  • 准确率奖励： 答案正确给高分，错误给低分。
  • 长度惩罚： 抑制无意义的冗长输出。
  • 步骤奖励： 对合理的中间推理或工具使用给予奖励。
  • 格式奖励： 输出满足指定结构时给分。

奖励函数必须与真实目标一致。
如果只奖励最终答案，模型可能学会跳过必要过程；如果过度奖励步骤数量，模型可能生成冗长但低价值的推理。

SFT 与 LoRA

SFT 的目标是让模型学会特定任务的标准响应方式。
在资源有限的情况下，通常不会全量更新所有模型参数，而是使用 LoRA 这类参数高效微调方法。

LoRA 的思路是在原模型权重旁边增加低秩适配矩阵，只训练少量新增参数。
这样可以降低显存和训练成本，也便于为不同任务维护不同适配器。

对智能体而言，SFT 适合学习：

  • 固定工具调用格式。
  • 标准任务分解方式。
  • 特定领域回答风格。
  • 基础推理轨迹。

但 SFT 本质上仍是模仿。
当示例数据覆盖不足时，模型很难自己发现更优策略。

GRPO 训练

GRPO 是第十一章的重点之一。
它与 PPO 同属策略优化思路，但 GRPO 使用组内相对奖励估计优势，减少对单独价值模型的依赖。

一个简化过程是：

  1. 对同一个问题采样多个回答。
  2. 分别计算每个回答的奖励。
  3. 用组内平均奖励作为基线。
  4. 奖励高于组均值的回答被强化。
  5. 奖励低于组均值的回答被抑制。

这种方法适合答案可以自动评分的任务。
例如数学题、代码题、格式化任务和可执行验证任务。

它的风险在于奖励函数一旦设计不当，模型会优化奖励漏洞，而不是真正提升任务能力。
因此训练过程中必须监控奖励、KL 散度、输出长度和错误类型。

评估与错误分析

训练完成不代表能力提升。
模型评估至少要覆盖准确率、格式合规率、推理步骤质量、工具调用成功率和失败类型分布。

错误分析比单一指标更有价值。
例如同样是数学题错误，可能来自：

  • 题意理解错误。
  • 公式选择错误。
  • 中间计算错误。
  • 输出格式错误。
  • 过早停止推理。

只有知道错误类型，才能判断该继续补充 SFT 数据、调整奖励函数，还是修改推理框架。

完整训练流程

一个相对完整的 Agentic-RL 流程包括：

  1. 准备任务数据集。
  2. 构造 SFT 数据和 RL 数据。
  3. 先进行 SFT，获得可用的基础行为。
  4. 设计奖励函数。
  5. 使用 GRPO 等方法进行强化学习。
  6. 在独立测试集上评估。
  7. 做错误分析和超参数调整。
  8. 合并或加载 LoRA 权重进行部署。

生产部署还要考虑量化、推理延迟、模型版本管理和回滚策略。
训练阶段的指标提升，不一定能直接转化为线上体验提升。

课后习题解析

SFT 与 RL 的关系

【原问题】

为什么不能只做 SFT，或者只做 RL？

【解析】

只做 SFT 的问题是模型只能模仿已有样本。
样本中没有覆盖的策略，模型很难通过监督学习获得。

只做 RL 的问题是探索空间过大，训练不稳定，且初始行为可能完全不符合任务格式。
尤其在工具调用场景中，模型如果连基本调用格式都不稳定，奖励信号会非常稀疏。

更合理的路线是先用 SFT 建立基础行为，再用 RL 针对任务结果做优化。

奖励函数的风险

【原问题】

奖励函数设计不当会产生什么问题？

【解析】

模型会优化奖励函数，而不是自动优化真实目标。

如果奖励只看答案是否包含某个数字，模型可能输出格式混乱但碰巧包含正确数字。
如果奖励过度惩罚长度，模型可能省略必要推理。
如果奖励过度鼓励步骤，模型可能生成很多无效中间过程。

因此奖励函数应尽量组合多个维度，并用人工抽查或独立评估集验证奖励是否真的代表任务质量。

第十二章 - 智能体性能评估

第十二章讨论智能体评估。
评估不是训练之后的附属动作，而是智能体工程化的基础设施。

没有评估，系统只能依赖主观体验判断是否变好。
一旦引入工具调用、检索、长程任务和多智能体协作，单看最终回答已经不够。

评估体系的基本问题

智能体评估需要回答四个问题：

  • 能不能正确理解任务。
  • 能不能选择合适工具。
  • 能不能完成多步推理。
  • 能不能在约束内给出可验证结果。

普通问答评估更关注文本正确性。
智能体评估还要关注行动过程，例如工具调用参数是否正确、是否错误调用工具、是否遗漏必要步骤、是否能处理工具失败。

BFCL：工具调用能力评估

BFCL（Berkeley Function Calling Leaderboard）主要评估模型的函数调用能力。
它关注模型能否根据用户意图生成正确的函数名和参数。

工具调用评估不能只看自然语言答案。
更关键的是：

  • 函数名是否匹配。
  • 参数名是否正确。
  • 参数值是否正确。
  • 参数类型是否正确。
  • 多函数调用顺序是否合理。
  • 并行调用和嵌套调用是否可执行。

BFCL 的评价重点

BFCL 适合评估模型接入工具层的基础能力。

例如用户要求“查询明天北京天气并设置提醒”，模型可能需要调用天气查询函数和提醒创建函数。
评估时不仅要看它是否知道要查天气，还要看参数是否被正确结构化。

工具调用错误通常比文本错误更危险。
文本回答错了，用户还能识别；工具参数错了，可能直接影响外部系统状态。

GAIA：通用助手能力评估

GAIA 更关注通用 AI 助手在真实任务中的综合能力。

这类任务往往需要检索、推理、工具使用和多步执行。
它不像 BFCL 那样只聚焦函数调用格式，而是评估智能体能否把多个能力组合起来解决问题。

GAIA 的难点

GAIA 的难点通常来自三类复杂性：

  • 信息来源复杂： 需要从多个来源查找和交叉验证。
  • 推理链路较长： 中间步骤错误会传导到最终答案。
  • 答案格式严格： 最终答案可能要求数字、字符串、列表或特定格式。

这类评估对长程上下文管理也有要求。
智能体需要记录已经查过什么、哪些信息可信、最终答案如何从证据中推出。

数据生成质量评估

第十二章还讨论了数据生成质量评估。
当智能体用于生成训练数据、数学题、测试样例或评测样本时，必须评估生成数据本身的质量。

常见方法包括 LLM Judge、Win Rate 和人工验证。

LLM Judge

LLM Judge 使用另一个模型作为裁判，对生成内容进行多维评分。

例如评估一道数学题时，可以从以下维度判断：

  • 题目是否清晰。
  • 难度是否合适。
  • 答案是否唯一。
  • 解法是否正确。
  • 是否与参考数据过度相似。

它的优势是成本低、速度快，适合大规模初筛。
缺点是裁判模型也会犯错，且可能偏向某些表达风格。

Win Rate

Win Rate 通过成对比较衡量质量。
系统把生成结果与基准样本或另一个模型结果放在一起，让裁判判断哪一个更好。

相比绝对打分，成对比较通常更稳定。
因为判断“A 是否优于 B”往往比直接给 A 打 8 分还是 9 分更容易。

但 Win Rate 也依赖样本选择。
如果基准样本质量本身不稳定，胜率就不能准确反映真实质量。

人工验证

人工验证成本最高，但在关键任务中不可替代。

尤其是数学题、医疗问答、金融建议、法律分析这类高风险内容，自动评估只能做初筛。
最终质量仍需要人工抽查，甚至需要专业人员审核。

评估报告与迭代

一个有用的评估报告不应只输出总分。
它应当能支持后续改进。

较完整的报告至少包括：

  • 总体指标。
  • 分类别指标。
  • 失败样本列表。
  • 错误类型分布。
  • 与历史版本的对比。
  • 典型失败案例。
  • 可复现实验配置。

评估的目的不是证明系统已经足够好，而是定位下一轮优化应该改哪里。
例如 BFCL 分数低，可能需要优化工具描述和参数 Schema；GAIA 分数低，可能需要改进检索、上下文管理或任务分解。

课后习题解析

为什么要分层评估

【原问题】

为什么不能只用一个综合指标评估智能体？

【解析】

综合指标能反映总体效果，但不能解释问题来源。

一个智能体任务失败，可能是检索失败、工具参数错误、推理错误、上下文污染或最终格式错误。
这些问题对应的修复方式完全不同。

因此评估体系应分层：

  • 工具调用能力用 BFCL 这类基准评估。
  • 通用多步任务用 GAIA 这类基准评估。
  • 生成数据质量用 Judge、Win Rate 和人工审核评估。
  • 线上系统还需要监控延迟、成本、失败率和人工接管率。

评估污染与可复现性

【原问题】

智能体评估中如何避免评估结果失真？

【解析】

评估失真常见于三类情况。

第一类是数据污染。
模型训练过程中可能已经见过评估集，导致分数虚高。

第二类是提示词过拟合。
系统为了提升某个榜单分数，针对评估集结构写了特殊提示，实际任务中并不能泛化。

第三类是实验不可复现。
模型版本、工具版本、搜索结果、随机种子和外部接口状态没有记录，后续无法解释分数变化。

因此评估时应保存完整配置，固定评估集版本，记录模型版本和工具版本，并保留失败样本。
对依赖外部搜索的任务，还应记录检索时间和检索结果摘要。
