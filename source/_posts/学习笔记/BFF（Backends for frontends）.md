## 问题背景

很多系统一开始并不会专门设计 BFF。前端调用后端接口，后端返回数据，页面展示结果，这种方式在项目早期通常没有问题。

问题通常出现在业务和端侧一起增长之后。后端接口越来越偏领域能力，例如订单、支付、物流、库存、售后；前端页面越来越偏用户任务，例如订单详情、个人中心、首页推荐、活动页。后端服务提供的是业务能力，前端页面需要的是展示结果，两者并不是同一种模型。

以订单详情页为例，页面可能需要展示订单基础信息、支付状态、物流进度、优惠信息、售后状态和可执行操作。如果前端直接调用多个后端接口，大致会变成这样：

```text
GET /orders/{id}
GET /payments?orderId={id}
GET /logistics?orderId={id}
GET /coupons?orderId={id}
GET /after-sales?orderId={id}
GET /orders/{id}/actions
```

这里的复杂度不只是“请求数量多”。前端还要处理多个接口的加载状态、错误合并、字段转换、状态拼装和兼容逻辑。页面越多，这些适配逻辑越容易散落在不同前端工程里。

另一类问题来自多端。Web 端可能需要完整字段和复杂操作入口，移动端可能只需要状态摘要，小程序可能更关注链路短和返回字段少。如果所有端都使用同一组通用接口，后端接口会不断膨胀；如果各端分别在前端处理适配，重复逻辑又会越来越多。

BFF 主要就是在这个背景下出现的。它要解决的不是“有没有后端”的问题，而是“前端体验层和后端领域层之间，是否需要一层稳定的服务端适配”。

## BFF 概述

BFF，全称是 Backends for frontends，意思是“面向前端的后端”。它不是具体框架，也不是固定组件，而是一种服务端分层方式。

它位于前端应用和后端服务之间，向上面向某一类前端体验提供接口，向下调用一个或多个后端服务。这里的“前端体验”可以是 Web 端、移动端、小程序，也可以是某个业务后台或某类页面。

常见链路可以简化成：

```text
前端应用 -> BFF -> 后端领域服务
```

存在统一入口层时，链路可能变成：

```text
前端应用 -> API Gateway -> BFF -> 后端领域服务
```

Sam Newman 对 BFF 的核心表述可以概括为：为特定用户体验创建对应的后端。这个“特定用户体验”很关键。BFF 不是为了给所有调用方再造一个通用后端，而是为了让某个前端或某类页面拥有更合适的服务端接口。

所以，BFF 的重点不是“后端 for 前端团队”，而是“后端 for 前端体验”。团队归属可以不同，但它的职责应当始终围绕前端体验展开。

## BFF 的作用

BFF 最直接的作用，是把前端页面需要的适配逻辑放到服务端集中处理。它向后端服务获取领域数据，再向前端返回更贴近页面的数据结构。

还是订单详情页的例子。引入 BFF 后，前端可以只调用一个页面接口：

```text
GET /api/order-detail/{id}
```

BFF 内部调用订单、支付、物流、优惠、售后等服务，然后返回面向页面的视图模型：

```json
{
  "orderId": "10001",
  "summary": {
    "statusText": "运输中",
    "amountText": "¥199.00"
  },
  "logistics": {
    "latestText": "包裹已到达杭州转运中心"
  },
  "actions": [
    {
      "code": "VIEW_LOGISTICS",
      "text": "查看物流"
    }
  ]
}
```

这个返回结果不是后端领域模型的原样暴露，而是面向订单详情页的视图模型。前端不需要知道支付服务、物流服务和售后服务分别如何组织数据，只需要消费页面需要的结果。

BFF 还可以隔离后端变化。后端服务拆分、字段调整、接口升级时，只要 BFF 对前端的返回结构保持稳定，前端就不需要跟着所有后端内部变化同步修改。

在单体迁移到微服务的过程中，BFF 也可以作为过渡层。前端仍然调用稳定的 BFF 接口，BFF 内部逐步从旧单体切换到新服务。这样可以减少迁移过程对前端的冲击。

## 适用场景

BFF 适合出现在前端和后端之间存在持续适配需求的地方。这里的“持续”很重要。如果只是一次性的接口不方便，通常不值得引入一层新的服务端应用。

比较典型的适用场景包括：

- 一个页面长期需要组合多个后端接口。
- 前端存在大量字段裁剪、字段转换和状态合并。
- Web、移动端、小程序之间展示差异明显。
- 后端接口偏领域模型，前端需要的是页面视图模型。
- 页面错误处理、缓存和降级需要贴近用户体验。
- 后端服务正在拆分，前端不适合直接感知所有服务变化。

### 页面模型和领域模型

页面模型服务于展示和交互，领域模型服务于业务规则和数据一致性。订单详情页需要“运输中”“待支付”“售后中”这类展示状态，但订单服务可能只关心订单状态，支付服务只关心支付状态，物流服务只关心物流轨迹。

把这些状态合并成页面展示结果，适合放在 BFF。把订单状态本身如何流转，放在 BFF 就不合适。

### 多端接口差异

多端差异明显时，BFF 能减少通用接口膨胀。Web 端、移动端、小程序不必强行消费同一个大而全接口，也不必在各自前端工程里重复处理大量字段裁剪。

这里有两种常见做法：

- 按端拆分 BFF，例如 Web BFF、Mobile BFF。
- 在同一个 BFF 中按页面或能力提供不同接口。

选择哪种方式取决于团队组织和端侧差异。如果不同端由不同团队长期独立演进，按端拆分更清晰。如果端侧差异不大，同一个 BFF 内部区分接口也可以接受。

### 失败降级

BFF 适合处理贴近页面体验的失败降级。例如订单详情页中，物流模块加载失败时，页面可以继续展示订单基础信息；优惠信息服务超时时，可以隐藏优惠模块或使用缓存。

这类逻辑不改变业务事实，只影响页面如何展示，因此放在 BFF 比放在领域服务中更自然。

## 不适用场景

BFF 不是所有项目都需要。它会增加一个服务端应用，也会带来新的部署、监控、测试和协作成本。

下面几类情况通常不适合引入 BFF：

- 只是做请求转发。
- 只是隐藏后端地址。
- 只是解决跨域问题。
- 项目规模很小，前端只调用少量接口。
- 后端接口本身已经贴近页面。
- 团队没有能力维护额外的服务端应用。

如果 BFF 只是把 `/api/user` 转发到后端 `/user`，它没有承担真正的适配职责。这类问题更适合由 API Gateway、反向代理或框架代理能力处理。

还有一种情况需要警惕：为了“架构完整”而引入 BFF。BFF 的价值来自明确的适配需求，不来自层数本身。没有稳定适配需求时，多一层只会增加复杂度。

## 职责边界

BFF 的边界决定了它能否长期维护。边界清楚时，它能降低前端复杂度；边界失控时，它会变成新的业务后端，而且往往比原来的后端边界更混乱。

BFF 适合承担的职责包括：

- 聚合多个后端接口。
- 裁剪字段，返回页面需要的视图数据。
- 适配不同端的展示差异。
- 统一前端可感知的错误结构。
- 处理轻量缓存和页面降级。
- 透传用户身份、租户、语言、地域等上下文。
- 隔离后端接口变化，维持前端接口稳定。

这些职责的共同点是：服务页面展示和交互，不决定核心业务事实。

### 核心业务规则

BFF 不应该实现核心业务规则。订单是否允许退款、退款金额如何计算、库存如何扣减、支付状态如何确认，这些都属于领域服务职责。

BFF 可以调用订单服务获取“是否允许退款”的结果，也可以把这个结果转换成页面按钮是否展示。但它不应该自己判断退款规则，更不应该直接修改订单状态。

可以用一个简单标准判断：如果这段逻辑决定了业务事实，就不应该放在 BFF；如果这段逻辑只是决定页面如何展示，可以考虑放在 BFF。

### 边界失控

BFF 边界失控时，常见表现包括：

- BFF 中出现大量业务状态变更逻辑。
- BFF 直接访问多个业务数据库。
- 同一条业务规则在 BFF 和领域服务中各实现一遍。
- 后端服务逐渐变薄，BFF 变成真正的业务系统。
- 修改业务规则时必须同步修改多个 BFF。

这些现象说明 BFF 已经不再是适配层，而是在侵入领域服务职责。

## 相关概念对比

BFF 经常和 API Gateway、聚合服务、普通后端接口混在一起讨论。它们确实有重叠，但关注点不同。

API Gateway 更偏基础设施层，负责统一入口治理，例如路由、鉴权、限流、日志、协议转换、灰度发布。BFF 更偏应用适配层，负责前端体验需要的数据组织，例如接口聚合、字段裁剪、视图模型转换和端侧差异适配。

聚合服务强调组合多个下游服务。BFF 也会聚合接口，但它的聚合目标更具体：服务某个前端体验。如果聚合结果同时服务多个系统和业务流程，并且包含稳定业务语义，它更接近通用聚合服务；如果聚合结果主要服务某个页面或某类端侧展示，它更接近 BFF。

普通后端接口通常表达业务能力，例如创建订单、查询库存、发起退款。BFF 接口通常表达页面需要，例如查询订单详情页展示数据、查询首页卡片列表、查询移动端个人中心摘要。

| 问题 | 更适合的位置 |
|---|---|
| 路由、限流、统一入口鉴权 | API Gateway |
| 页面数据聚合、字段裁剪 | BFF |
| 订单是否允许退款 | 订单服务 |
| 用户展示名称格式化 | BFF |
| 库存扣减 | 库存服务 |
| 多端返回字段差异 | BFF |
| 支付状态流转 | 支付服务 |
| 前端错误信息归一化 | BFF |

## 引入成本

引入 BFF 不是纯收益。每增加一个 BFF，就增加一个需要维护的服务端应用。

工程上，它需要部署、监控、告警和故障处理；调用后端服务时，还要处理超时、重试、熔断和降级。协作上，前端、BFF、后端服务之间会形成新的接口契约。维护上，BFF 容易因为需求快速变化而不断膨胀。

复用也是一个需要克制的问题。多个 BFF 之间可能会出现重复代码，但不能为了消除重复就抽出过重的共享层。过度共享会让多个前端体验重新耦合在一起，最后又回到通用 API 的问题。

比较稳妥的做法是：先允许少量重复，只有当重复逻辑稳定、边界清晰、与前端体验无关时，再考虑抽成共享库或下沉到领域服务。

## 总结

BFF 是面向前端体验的服务端适配层。它适合处理接口聚合、字段裁剪、视图模型转换、端侧差异、轻量缓存、降级和错误归一化。

是否引入 BFF，关键不在于项目有没有前端，而在于前端和后端之间是否存在持续、稳定、明确的适配需求。

BFF 的边界必须清楚：页面展示和交互适配可以放在 BFF，核心业务规则必须留在后端领域服务。边界清楚时，BFF 能降低前端复杂度；边界失控时，BFF 会变成新的业务复杂度来源。
